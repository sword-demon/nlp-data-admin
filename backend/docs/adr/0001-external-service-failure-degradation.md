# ADR 0001 — 外部服务失败不得击穿核心创作链路

- **日期**：2026-05-01
- **状态**：Accepted
- **上下文级别**：Backend

## 背景

创作工坊依赖多个外部服务作为"能力增强"：

- **AI 模型**（DashScope / OpenAI）：标题、大纲、正文、配图分析
- **配图源**（Pexels / Iconify / Mermaid.ink / DashScope 万相）：插图获取
- **搜索服务**（Exa；未来可能接入 Tavily）：选题背景研究（新增于 2026-05）

这些服务各自有独立的 SLA、速率限制、鉴权要求。任何一个挂掉（429 限流、网络抖动、API Key 过期、用户未配置）都可能导致用户整个创作流程被阻塞。

## 决策

**外部服务分两类，各自的失败处理规则截然不同：**

| 类别         | 代表                        | 失败处理                                                                      | 用户可见性               |
| ------------ | --------------------------- | ----------------------------------------------------------------------------- | ------------------------ |
| **核心能力** | 标题/大纲/正文/配图分析 LLM | 硬失败：`markFailed(article)` + `WORKSHOP_AGENT_FAILED`                       | 明确报错，用户重试       |
| **增强能力** | 研究搜索、配图源、AI 生图   | 降级继续：对应字段置 null/空，流程正常流转；响应带布尔提示 `*_fallback: true` | 静默或 toast，不打断流程 |

### 具体规则

1. **状态机不绕行**：即使触发降级，状态仍按完整路径流转（如 `TOPIC_RESEARCHING → TITLE_GENERATING`），不能跳过状态节点——保持历史轨迹完整。
2. **降级必写 AgentLog**：降级不是"静默吞掉"。`AgentLog` 里必须有一条 `status=failed` 记录供可观测性面板查失败率。
3. **配额不二次扣费**：降级是系统内部行为，用户不为外部服务不稳定买单，`QuotaService` 每次创作只扣一次。
4. **API Key 缺失 = 显式降级**：`EXA_ENABLED=false` / `API_KEY` 为空时**不发网络请求**，直接降级路径。
5. **响应契约**：降级发生时返回体中带 `research_fallback: true`（或 `image_fallback: true`），前端可选展示。不返回具体原因字符串（避免后端实现细节泄漏）。

## 考虑过的备选

- **硬失败 + 回滚 article**：删除刚创建的 article，返回 500。否决理由：用户对"锦上添花"的步骤失败敏感度低，为此阻塞主流程不合算。
- **硬失败 + 保留 article = FAILED**：article 留在列表里。否决理由：列表被废记录污染，且研究失败不等于创作无法进行。
- **每个外部服务独立配额**：给用户独立的 Exa 配额、Pexels 配额。否决理由：用户心智模型复杂，难以理解。

## 影响

- 新增 Agent / 新增外部 provider 时必须明确归类（核心 vs 增强），并实现对应失败分支。
- `QuotaService::consumeQuota` 语义固化为"每次 `startCreation` 成功扣一次"，后续不要在子步骤里再加扣费逻辑。
- 可观测性面板需要按 `agent_name + status` 聚合，暴露"静默降级率"——否则降级变成黑箱。

## 首批应用

- `Pexels/Nanobanana` 配图：已按增强能力处理（`*_ENABLED` 开关 + 策略工厂降级）。
- `TopicResearchAgent`（本 ADR 触发）：新增 Exa 研究属于增强能力，失败降级，响应带 `research_fallback: true`。
